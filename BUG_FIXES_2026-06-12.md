## A. Broken core functionality

| # | Sev | Issue | Root cause | Fix |
|---|-----|-------|-----------|-----|
| A1 | 🔴 | **Saving a quote always failed (HTTP 500)** — quote creation was completely broken in production | The quote form posts helper fields (`contact_name`, `contact_email`, `project_type`, `project_address`) that don't exist as columns in the `quotes` table; Postgres rejected every insert | `POST/PATCH /api/quotes` now whitelist real table columns and normalize empty-string dates/UUIDs to `null` (`app/api/quotes/route.ts`, `app/api/quotes/[id]/route.ts`) |
| A2 | 🔴 | **Invoice draft failed (HTTP 500) when due date left blank**; every failed attempt silently left behind an orphaned auto-created "Invoice Job" project (20+ strays found and removed) | Postgres rejects `""` as a `date` value; the auto-project is created before the invoice insert | Empty-string date/UUID normalization in `POST/PATCH /api/invoices`; stray projects deleted |
| A3 | 🔴 | **Chatbot always errored** ("The assistant ran into an issue…") | `gemini-1.5-flash` was retired by Google — API returns 404 (verified with your key) | Switched to `gemini-2.5-flash` (tested working), raised token budget for the thinking model, capped input size (12 messages × 2,000 chars) |
| A4 | 🔴 | **"Application error: a client-side exception…" on Scheduling, Notifications, and any page showing an empty state** | `EmptyState` in `components/ui.tsx` used `typeof icon === "function"`, but Lucide icons are `forwardRef` objects — the raw object was rendered as a React child (React error #31) | Rewritten with `isValidElement` check |
| A5 | 🟠 | **Stripe subscription didn't activate the plan after checkout** | Post-checkout sync required a `stripe_subscription_id` that only the (unconfigured) webhook would have stored — chicken-and-egg | Success URL now carries `session_id={CHECKOUT_SESSION_ID}`; `/api/billing/sync` resolves subscription + customer from the checkout session and upserts plan/cycle/status/period — works with no webhook at all |
| A6 | 🟠 | **Paying customers got locked out when their original trial date passed** | `checkTrialAccess` only looked at `businesses.trial_ends_at`; nothing updated it on subscribe | An active/trialing/past_due paid subscription now always grants access |
| A7 | 🟠 | **Webhook missed initial subscriptions and could mis-handle renewals** | Handler only processed `customer.subscription.updated`; newer Stripe API moved `current_period_end` onto the subscription item | Now also handles `customer.subscription.created`, reads period-end from both locations, upserts by `business_id`, and invoice payments are idempotent via `payment_intent_id` |
| A8 | 🟠 | **Sidebar plan timeline always showed "14 days left"** regardless of plan | Computed from `subscription.created_at` with hardcoded 14/15 totals; ignored the plan entirely | Plan-aware badge: paid plans show "Pro plan · Renews <date> · Manage plan"; trials count down from the real `trial_ends_at` (EN/ES/PT strings added) |
| A9 | 🟡 | Trial length inconsistent — DB says 14 days but UI/chatbot said "15 days" | Copy drift | Standardized to 14 days (subscription page, progress bar, chatbot prompt) |

## B. Security

| # | Sev | Issue | Fix |
|---|-----|-------|-----|
| B1 | 🔴 | **Middleware made every route public** — `"/"` was in `PUBLIC_PATHS` and matched with `startsWith`, so all paths were "public" (caught in uncommitted local code before it shipped) | `"/"` is now an exact match; default-deny preserved for everything else |
| B2 | 🟠 | **Static files were auth-redirected** — `/logo.png` returned the login page HTML, breaking the logo app-wide (`/_next/image` 400 on every page) | Middleware matcher excludes any path containing a file extension |
| B3 | 🟠 | **Mass assignment in `POST /api/businesses`** — raw request body spread into the insert; a user could set `stripe_customer_id`, `trial_ends_at`, `restricted_mode` | Strict field whitelist |
| B4 | 🟠 | **14 create/send API routes had no trial gate** — expired-trial users could still create projects/tasks/templates/updates/item-requirements, send communications/notifications/invoices, invite team members, duplicate templates, snapshot quote versions, request feedback | `checkTrialAccess` added to all of them; support tickets and business creation deliberately remain open |
| B5 | 🟠 | **No rate limiting on auth endpoints** — brute force possible | In-memory fixed-window limiter: login 10/5min, register 5/15min, forgot-password 5/15min, reset-password 10/15min (`lib/rate-limit.ts`) |
| B6 | 🟡 | No minimum password strength on register/reset | 8-character minimum enforced server-side |
| B7 | 🟡 | Cross-tenant contact lookups in quote-send and invoice-send email paths (contact fetched by id without `business_id`) | Both scoped to the session business |
| B8 | 🟡 | Audit log visible/accessible to all team roles | Owner/admin only: API returns 403, sidebar + mobile More hide the link, page shows an access notice |
| B9 | 🔵 | Chatbot had no input limits (unbounded paid-API usage) | History capped at 12 messages, 2,000 chars each |

Also shipped (pre-existing local hardening reviewed and included in the deploy):
JWT and admin-JWT secrets now required (no dev fallback), `force_logout_at`
session invalidation on password change/reset and team removal, settings field
whitelist, impersonation-exit verification, the Playwright test-setup endpoint
returns 404 in production, payments invoice/contact lookups business-scoped.
